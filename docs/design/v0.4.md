# rspyts 0.4: direct Rust API compilation

Status: draft for implementation

rspyts 0.4 is a clean-slate architectural replacement. It compiles a curated
Rust API into self-contained Python and TypeScript packages. Rust remains the
only authored contract; generated host code is an ignored build artifact.

The release is intentionally called `0.4.0`, not `4.0.0`. The project is still
pre-1.0, and this design deliberately replaces the 0.3 contract rather than
preserving it through compatibility layers.

The implementation starts from an empty product skeleton in the existing Git
repository. The v0.3 implementation is history and evidence, not a foundation.
The 0.4 branch may delete every existing crate, host runtime, example, test,
workflow, and document whose purpose does not follow from this RFC. Git tags
preserve released v0.3 source; the new implementation does not preserve it.

## The problem

rspyts 0.3 proves that one Rust implementation can run in Python and
TypeScript, but it puts too much integration work in consuming repositories:

- consumers often mirror existing domain models as `Bridge*` transport DTOs;
- consumers maintain conversions between the mirrors and real Rust types;
- generated Python and TypeScript implementation files are committed;
- generated clients depend on separately released rspyts host runtimes;
- projects maintain their own WASM staging and packaging scripts;
- drift is detected by comparing generated text rather than the API's meaning.

The Neurovirtual Cloud migration makes the cost concrete. Hardware and
Evaluation currently contain thousands of lines of handwritten bridge DTOs and
conversions even though the corresponding Rust domain types already exist.
That is managed duplication, not one source of truth.

0.4 changes the unit of abstraction. It does not generate a toolkit from which
an application authors another bridge. It builds the host package.

## Outcome

A consumer authors this:

```rust
use rspyts::{Error, Type};

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize, Type)]
#[serde(rename_all = "camelCase", deny_unknown_fields)]
pub struct Summary {
    pub count: u64,
    pub average: f64,
}

#[derive(Debug, thiserror::Error, Error)]
pub enum SummaryError {
    #[error("values cannot be empty")]
    Empty,
}

#[rspyts::export]
pub fn summarize(#[rspyts(buffer)] values: &[f64]) -> Result<Summary, SummaryError> {
    // The real domain function. There is no bridge version of Summary.
    todo!()
}

rspyts::module!();
```

The build produces:

```text
.rspyts/
  contract.json
  python/
  typescript/
  artifacts/
```

`.rspyts/` is always ignored. Its Python output, native extension, TypeScript
output, JavaScript glue, declarations, and WASM are copied into wheels and npm
packages during packaging. None is checked into the source repository.

The installed packages expose ordinary domain names such as `Summary` and
`summarize`. `BridgeSummary`, `bridge_summarize`, generated public facades, and
private-prefix naming conventions are not part of the model.

## Non-negotiable rules

An implementation is not 0.4 if any of these rules is false:

1. Existing Rust domain types and functions are the contract.
2. A consumer does not create a structurally equivalent DTO for an exportable
   Rust type.
3. Generated implementation source exists only below `.rspyts/`, a temporary
   build directory, or a package artifact.
4. A shipped wheel or npm package has no runtime dependency on a Python or npm
   rspyts package.
5. Python uses PyO3 with the stable ABI; TypeScript uses wasm-bindgen. These are
   implementation details hidden by rspyts.
6. One semantic IR drives types, calls, errors, constants, documentation, and
   both backends.
7. The compiler rejects a type it cannot describe exactly. It never silently
   widens a type, re-emits a foreign type under a new identity, or guesses a
   custom Serde shape.
8. No v0.3 compatibility aliases, facade modules, ABI adapters, or dual paths
   are added to v0.4.
9. No v0.3 implementation code is reused. A third-party dependency may be
   selected again only from a fresh evaluation against the 0.4 requirement;
   no internal module is copied, moved, or adapted forward.

## Clean-slate rule

0.4 is not a refactor of `rspyts-core`, the ABI-3 envelope, the ctypes runtime,
the TypeScript runtime, the v0.3 emitters, or their tests. Those components are
deleted at the start of implementation.

The old repository may be consulted to recover requirements and failure cases.
Code is then written against the 0.4 IR and acceptance corpus. A previous test
is recreated only when it describes behavior required by this RFC; it is not
ported merely to retain coverage statistics.

Repository identity, license, release history, and published tags remain.
Implementation structure, dependencies, CLI compatibility, generated layouts,
and runtime architecture do not.

## What remains generated

Generated code is still useful. Committing it and making applications maintain
it is not.

The Python build may generate Pydantic models, typed functions, exceptions,
resource classes, codecs, and type information. The TypeScript build may
generate interfaces, unions, functions, resource classes, declarations, and
WASM initialization code. Backend glue may be substantial. It belongs to
rspyts and is tested there.

This distinction is important:

- **authored source:** Rust domain API and small rspyts annotations;
- **reviewed metadata:** an optional semantic contract lock;
- **ignored build output:** all generated Python, TypeScript, Rust adapter,
  JavaScript, schema, native, and WASM files;
- **published artifact:** the self-contained wheel or npm package.

Generated implementation size therefore does not count against the consuming
repository, but it does count against rspyts's own maintainability and test
burden.

## Public Rust surface

0.4 has four concepts.

### `Type`

`#[derive(rspyts::Type)]` marks an existing value type for use in exported
signatures. The derive reads the type's Rust shape, supported Serde attributes,
documentation, and rspyts transport hints.

It does not create a second public Rust type.

The normal path is deliberately boring:

```rust
#[derive(serde::Serialize, serde::Deserialize, rspyts::Type)]
#[serde(rename_all = "snake_case", deny_unknown_fields)]
pub struct SignalDefinition {
    pub id: SignalId,
    pub kind: Option<SignalKind>,
    pub backing: SignalBacking,
}
```

### `Error`

`#[derive(rspyts::Error)]` marks an existing Rust error enum. Variant identity,
message, and structured fields become typed host exceptions. A project should
export its domain error when that is the intended public contract. It may
define one narrow API error enum when multiple internal errors must be
intentionally coarsened, but it must not duplicate data models to do so.

### `export`

`#[rspyts::export]` marks an existing public function, constant, or impl member.
It preserves the Rust item and generates backend registration and metadata.

Target selection is allowed only where the operation itself is not portable:

```rust
#[rspyts::export(python)]
pub fn open_path(path: String) -> Result<Recording, Error> {
    // Native file IO.
}
```

Target selection does not justify different data contracts for each language.

### `module!`

`rspyts::module!()` appears once in the exporting crate. It closes the module,
validates duplicate exports, and provides the compiled semantic manifest used
by the compiler and the two backends.

## Honest escape hatches

Some valid domain types do not expose a shape a derive can infer. 0.4 handles
these cases on the existing type instead of requiring a mirror.

### Declared wire shape

A type with custom validation or custom Serde code declares the shape that its
existing serializer actually exposes:

```rust
#[derive(serde::Serialize, serde::Deserialize, rspyts::Type)]
#[serde(transparent)]
#[rspyts(wire = String)]
pub struct AnnotationId(String);

#[derive(rspyts::Type)]
#[rspyts(wire = Option<f64>)]
pub enum MetricValue {
    Number(f64),
    NotApplicable,
}
```

The normal Serde implementation remains the conversion and validation
boundary. rspyts uses `wire` only to describe and validate the host contract.
The compiler type-checks the Serde implementation, and generated decoders
validate actual values against the declared shape. A consumer adds one Rust
round-trip fixture for each special wire declaration because arbitrary custom
Serde code cannot be proven correct by a derive macro.

### Checked scalar projection

Machine-sized or otherwise non-portable scalars require an explicit portable
projection:

```rust
pub struct ChannelBacking {
    #[rspyts(as = u64)]
    pub source_index: usize,
}
```

The generated conversion is checked in both directions. Overflow is a typed
boundary error.

### Transport policy

Efficient transport is attached to an existing field or parameter. It does not
replace the domain type:

```rust
pub struct Chunk {
    #[rspyts(buffer)]
    pub values: Vec<f64>,
}

#[rspyts::export]
pub fn parse(#[rspyts(bytes)] input: &[u8]) -> Result<Recording, Error> {
    todo!()
}
```

`buffer` maps numeric Rust storage to NumPy arrays and JavaScript typed arrays.
`bytes` maps byte storage to Python `bytes` and `Uint8Array`. Ownership and
copying are explicit IR properties.

Stateful domain types are exported through their existing impl rather than a
wrapper struct:

```rust
#[rspyts::export]
impl Recording {
    #[rspyts(constructor)]
    pub fn open_bytes(#[rspyts(bytes)] bytes: Vec<u8>) -> Result<Self, Error> {
        todo!()
    }

    pub fn manifest(&self) -> &RecordingManifest {
        todo!()
    }
}
```

The generated hosts own an opaque resource. The Rust type, methods, and
validation remain the implementation.

### Manual contract implementation

`rspyts::Type` may be implemented manually only when `wire`, `as`, `buffer`, or
`bytes` cannot express an honest contract. The implementation must supply a
semantic shape and checked conversions. Cloud acceptance requires an explicit
review of every manual implementation; none may be introduced merely to keep a
legacy host API unchanged.

These are shape annotations, not parallel DTOs. The expected Cloud rule is
zero `Bridge*` types and very few manual `Type` implementations.

## Semantic IR

The compiled manifest is the single intermediate representation for every
backend. It records:

- stable item and type identities based on Cargo package plus Rust module path;
- structs, enums, newtypes, fields, presence, nullability, and wire names;
- functions, methods, constructors, constants, errors, and target scope;
- scalar widths and signedness;
- buffer dtype, ownership, and mutability;
- resource/handle lifecycle;
- documentation and deprecation metadata;
- defining package and cross-package host import identity;
- compiler schema version and a canonical contract fingerprint.

The IR describes semantics, not a shared byte protocol. Python and
TypeScript may use different safe transports while exposing the same declared
values and behavior.

The compiler validates the complete graph before generating anything. Cycles,
unsupported generic shapes, ambiguous Serde behavior, duplicate host names,
unresolved foreign types, and target-inconsistent contracts are hard errors.

## Compiler pipeline and repository shape

The clean implementation uses three Rust crates:

| Crate | Responsibility |
|---|---|
| `rspyts` | Public derives/attributes, semantic IR, conversion traits, and target-gated backend support |
| `rspyts-macros` | Proc-macro implementation, separate because Rust requires it |
| `rspyts-cli` | Contract loading, validation, emitters, build orchestration, locking, and inspection |

There are no Python or npm runtime packages. A small Python distribution may
provide the build hook, but it is installed only in an isolated build
environment and is not imported by the resulting wheel.

`rspyts build` performs one deterministic pipeline:

1. Compile a contract probe from the actual crate, features, and `cfg` set.
2. Load and validate the semantic IR exported by `rspyts::module!()`.
3. Render host source and metadata into a fresh staging directory.
4. Build the PyO3 native extension and/or wasm-bindgen module from the same
   annotated Rust crate.
5. Verify that source, declarations, executable artifacts, and dependency
   contracts carry the same fingerprint.
6. Atomically replace `.rspyts/` and report the artifact paths as stable JSON.

The manifest export is a build-time inspection surface, not a call ABI.
Application calls never cross the generic C envelope used by 0.3.

## Type system for 0.4

The initial release supports the shapes Cloud uses today:

| Rust | Python | TypeScript |
|---|---|---|
| `bool` | `bool` | `boolean` |
| fixed-width integers through 32-bit | checked `int` | checked `number` |
| `i64`, `u64` | checked `int` | `bigint` |
| finite `f32`, `f64` | `float` | `number` |
| `String`, borrowed string input | `str` | `string` |
| `&T` input | decoded `T`, borrowed for the call | decoded `T`, borrowed for the call |
| shared borrowed value return | copied host value | copied host value |
| `Option<T>` | `T | None` | `T | null` |
| `Vec<T>` | `list[T]` | `T[]` |
| string-keyed maps | `dict[str, T]` | `Record<string, T>` |
| fixed tuples | typed tuple | typed tuple |
| value structs | Pydantic model | interface |
| unit and tagged enums | enum/models | union |
| `serde_json::Value` | JSON-compatible value | `unknown` |
| numeric buffer policy | NumPy array | typed array |
| byte policy | `bytes` | `Uint8Array` |
| exported stateful type | resource class | disposable resource class |

`i128`, `u128`, unconstrained generics, callbacks, foreign borrowed lifetimes,
interior foreign references, and arbitrary custom serializers are rejected.
`usize` and `isize` require an explicit fixed-width projection.

A Rust function may return `&T`, `&str`, or a borrowed slice when the complete
value is copied or serialized before the host call ends. The host never
receives a reference into Rust memory.

Structured floats are finite. Numeric buffers preserve IEEE values. Exact
64-bit values remain native Rust integers and project to TypeScript `bigint`;
there is no application-authored decimal-string wrapper.

0.4 remains synchronous. Async calls, streams, callbacks, and cancellation are
deferred until Cloud has a real use case and an ownership design. CPU-heavy
browser calls run in a Web Worker. This keeps the first release focused on the
actual Hardware, Evaluation, and Platform requirements.

## Python backend

The Python backend is implemented on PyO3 and targets CPython's stable ABI.
rspyts owns the PyO3 code generation; consuming projects do not write or see
it.

The built wheel contains:

- the native extension;
- generated public models, enums, functions, errors, and resource classes;
- type information and `py.typed`;
- embedded backend helpers needed by that package.

The wheel may depend on normal host libraries selected by the contract, such
as Pydantic and NumPy. It does not depend on the PyPI rspyts distribution at
runtime. rspyts appears only in the build-system requirements.

PyO3 owns module loading, the stable CPython boundary, panic conversion, and
native object lifetime. rspyts owns semantic conversion, contract validation,
buffer policy, generated public types, and typed errors.

Resource classes support `close()` and context managers. Closing twice is
safe. Use-after-close is a typed error. A finalizer is only a leak backstop.

## TypeScript backend

The executable TypeScript backend is implemented on wasm-bindgen. The npm
artifact contains:

- generated JavaScript and TypeScript declarations;
- the WASM module or a package-declared WASM asset;
- generated initialization, functions, types, errors, and resource classes;
- embedded helpers required by that package.

It has no npm runtime dependency on rspyts. wasm-bindgen owns WebAssembly
imports, exports, memory interaction, and standard JS glue. rspyts owns the
semantic API, generated declarations, checked values, buffer policy, errors,
and resource lifecycle.

Resource classes support `free()` and `Symbol.dispose`. A trapped or panicked
instance is poisoned and cannot be reused.

The browser is the required TypeScript runtime. Node may load the same WASM,
but 0.4 does not add a separate Node native backend.

## Static TypeScript mode

Some crates, such as Platform, need Rust-owned vocabulary in TypeScript but do
not need to execute Rust in the browser. The TypeScript target therefore has a
static mode for types, enums, constants, and data tables.

Static mode does not transpile arbitrary Rust functions. A classifier that
must remain static is represented by exported data and a small host operation,
or it uses WASM. rspyts will not claim to translate Rust logic that it cannot
prove equivalent.

Static output is generated below `.rspyts/` and included in the npm `dist`
directory. It is not committed.

## Cross-crate types

Foreign contract types retain the identity of their defining Rust crate. Their
Cargo metadata declares canonical host packages:

```toml
[package.metadata.rspyts]
python = "neurovirtual.hardware"
typescript = "@neurovirtual/hardware"
```

When Evaluation exposes Hardware's `Annotation`, generated Evaluation code
imports `neurovirtual.hardware.Annotation`. It does not emit another Annotation
model. Dependency contract fingerprints are embedded in the package and
checked when loaded.

An unresolved foreign type is a build error. A package may explicitly request
embedding for a deliberately standalone artifact, but that is never the
default and must use the defining type's stable semantic identity.

## Build and packaging

The normal command set is small:

```text
rspyts build          compile the contract and every configured backend
rspyts check          build clean artifacts and run semantic/conformance checks
rspyts lock           accept the current semantic public contract
rspyts inspect        print the compiled contract and fingerprints
rspyts clean          remove .rspyts
```

There is no `generate` command because generated source is not a source-control
operation. There is no project-owned WASM build script.

Configuration selects the crate, host package identities, and backend mode. It
does not ask consumers to choose generated source directories:

```toml
[crate]
path = "rust"
features = ["bindings"]

[python]
package = "neurovirtual.hardware"

[typescript]
package = "@neurovirtual/hardware"
mode = "wasm"
```

All paths resolve from `rspyts.toml`. Output always goes below the sibling
`.rspyts/` directory unless a packaging process supplies a temporary staging
directory.

Python packaging uses a build-only rspyts hook. TypeScript packaging invokes
`rspyts build` before the existing package compiler and consumes its reported
staging directory. The rspyts build report is stable JSON, so Moon, Cargo,
Hatch, and npm do not need custom discovery scripts.

## Divergence prevention

0.4 prevents three different forms of drift.

### Rust versus generated hosts

Every host artifact is generated from the freshly compiled IR. Its public
types and executable artifact embed the same contract fingerprint. Packaging
fails if the fingerprints differ.

Because generated files are never edited or committed, there is no stale
generated-source state to reconcile.

### Python versus TypeScript

Both backends consume the same IR and must pass the same backend-neutral
contract corpus. Per-type semantic fingerprints must match. Backend-specific
representations such as Python `int` and TypeScript `bigint` are declared IR
mappings, not independent application decisions.

### Current API versus reviewed API

`rspyts lock` writes one optional, compact `rspyts.lock` containing semantic
item signatures and hashes. It is metadata analogous to `Cargo.lock`, not
generated implementation source. `rspyts check --locked` reports additions,
removals, and compatible or breaking changes by item and path.

Cloud commits this lock for Hardware, Evaluation, and Platform. Updating it is
the explicit review step for an intentional public contract change.

## Testing ownership

rspyts owns exhaustive bridge correctness:

- primitive boundaries and invalid values;
- nested structs, enums, options, maps, and tuples;
- exact integer ranges;
- finite structured floats and raw buffer floats;
- custom declared wire shapes;
- bytes and every numeric buffer dtype;
- errors and panics;
- resource construction, use, close/free, stale access, and finalization;
- cross-crate type identity and fingerprint mismatch;
- deterministic artifacts and ignored-output policy;
- real wheel installation and native calls;
- real browser WASM loading and calls.

Property tests run the same semantic values through Rust, Python, and
TypeScript. Backend tests compare values and errors, not rendered source text.

Consumers own domain behavior, a small package smoke test, and their public API
review. They do not repeat rspyts codec and lifecycle tests.

## Deliberate non-goals

0.4 does not:

- export arbitrary Rust automatically;
- parse or translate arbitrary Rust function bodies;
- support every Serde customization;
- provide a generic language-neutral C ABI;
- provide a plugin backend system;
- offer multiple Python model frameworks;
- support callbacks, async, streams, or borrowed foreign references;
- preserve v0.3 generated APIs or runtime packages;
- make private or unstable Rust internals public merely because they compile.

The closed surface is a feature. A rejected API should usually be reshaped as a
better public Rust API, not handled by adding another generic escape mechanism.

## Cloud acceptance gates

Cloud is the acceptance corpus, not a place to finish the compiler by hand.
The migration may proceed only when all gates below pass.

### Evaluation vertical slice

- `bridge_contracts.rs` is deleted.
- Evaluation exports its existing policy, request, result, event, and metric
  types directly.
- Hardware Annotation is imported under its Hardware host identity.
- Generated Python source is absent from Git.
- The wheel installs and runs Evaluation through PyO3 with no rspyts runtime
  dependency.

Evaluation is the first slice because it exercises a large value graph and
cross-crate types without requiring WASM resources.

### Hardware release gate

- every `bridge_contracts/` module is deleted;
- no `Bridge*` data model or conversion graph remains;
- existing validated identifiers and domain models use direct derives plus
  truthful wire-shape hints;
- existing `Vec<T>` and byte storage uses transport annotations rather than
  `Buf`/`Bytes` replacement DTOs;
- the Python wheel uses PyO3 and the TypeScript package uses wasm-bindgen;
- the custom `build-wasm.mjs` path is deleted;
- generated files are ignored and absent from public declaration audits;
- native path operations remain Python-only, while portable byte operations
  behave equivalently in Python and a real browser;
- package artifacts contain matching contract fingerprints;
- Hardware's lock reports no unreviewed public contract change.

Hardware is the final 0.4 release gate because it exercises buffers, bytes,
large value graphs, resources, target-specific IO, errors, Python, and browser
WASM.

### Platform static gate

- Rust remains authoritative for event, metering, and identity vocabulary;
- TypeScript types, constants, and tables are generated only during build;
- arbitrary classifiers are not falsely transpiled;
- Python native behavior and TypeScript static data carry matching semantic
  fingerprints;
- no committed bindings or rspyts runtime dependencies remain.

### Repository-wide gate

```text
rg 'Bridge[A-Z]|bridge_contracts' packages/{hardware,evaluation,platform}
```

must find no contract mirror. Narrow error names or documentation discussing
the migration require explicit review; generated directories are not exempt
because they must not be tracked at all.

## Release strategy

0.4 is implemented from a blank product skeleton, and Cloud is migrated from a
fresh current worktree. Neither codebase is incrementally patched from the old
experimental bridge.

1. Preserve the v0.3 tags, then delete the v0.3 implementation on the 0.4
   branch.
2. Freeze the 0.4 IR and macro vocabulary in this RFC.
3. Implement a minimal compiler with static output and semantic locking.
4. Implement the PyO3 backend and migrate the Evaluation vertical slice in an
   isolated Cloud worktree.
5. Implement the wasm-bindgen backend and migrate a narrow Hardware recording
   slice.
6. Complete Hardware and pass its release gate.
7. Migrate Platform static data.
8. Only then rebuild the complete Cloud migration from current `origin/ultra-clean`.
9. Release rspyts 0.4.0 and update Cloud's build dependencies in one reviewed
   change.

The 0.3 releases remain available through their tags and registries. There is
no 0.3 maintenance requirement, mixed 0.3/0.4 package graph, or compatibility
layer. A Cloud package moves only when its full 0.4 target artifact is ready.

## Definition of success

The release succeeds when a Cloud maintainer can change an exported Rust field,
run one build, and receive matching Python and TypeScript packages or one clear
semantic error—without editing a mirror type, conversion, generated source,
runtime dependency, or WASM build script.

That is the purpose of rspyts 0.4. Everything else is implementation detail.
