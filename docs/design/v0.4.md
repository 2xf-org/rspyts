# rspyts 0.4 architecture

Version 0.4 is a clean-slate compiler for a curated Rust API. The existing
Rust types and functions are the only authored contract. Python and TypeScript
representations are generated build output.

This document is normative for ownership and architecture. User-facing syntax
and mappings live in the [reference](../reference.md).

## Why 0.4 exists

The 0.3 design could execute one Rust core in multiple hosts, but consumers
still authored transport DTOs, conversion graphs, generated-source facades,
and packaging scripts. That prevented accidental drift while retaining the
duplicate surfaces that could drift.

Version 0.4 changes the unit of work: it builds a host package from the domain
API instead of generating a toolkit for writing a second API.

## Invariants

An implementation is not rspyts 0.4 if any invariant below is false:

1. The existing Rust domain API is the authored source of truth.
2. Consumers do not create structurally equivalent `Bridge*` contracts.
3. All generated implementation and compiled staging files live below
   `.rspyts/`, a temporary directory, or a published consumer artifact.
4. Generated source is ignored; the semantic `rspyts.lock` is reviewed.
5. A shipped consumer package has no Python or npm rspyts dependency.
6. Python calls Rust through PyO3's stable ABI; executable TypeScript calls
   Rust through wasm-bindgen.
7. Static TypeScript emits vocabulary and data, not translated function bodies.
8. The compiler rejects any contract it cannot describe without loss.
9. There is no generic C application-call protocol.
10. There is no 0.3 compatibility layer or reused 0.3 implementation path.

## Product crate shape

The implementation has exactly three publishable Cargo crates; acceptance
fixtures may contain unpublished consumer crates:

| Crate | Responsibility |
| --- | --- |
| `rspyts` | Public macros, closed semantic IR, registry, boundary values, and target support |
| `rspyts-macros` | Macro parsing and lowering; separate because proc macros require it |
| `rspyts-cli` | Cargo contract loading, validation, fingerprints, locks, emitters, and build orchestration |

There is no `rspyts-core`, Python rspyts distribution, npm rspyts package, or
host runtime release.

## One lowering, two responsibilities

The macros lower each annotated Rust item once and emit:

- deterministic semantic registration used by the compiler; and
- target-gated PyO3 or wasm-bindgen wrappers adjacent to the real Rust item.

The compiled semantic manifest drives validation, host source, declarations,
static TypeScript, fingerprints, locks, and packaging metadata. It does not
generate Rust wrappers in a second source-generation pass, and it is not a
generic call envelope.

The build-only manifest inspection surface exists so the CLI can load the
contract produced under the consumer's real Cargo features and `cfg` set.
Application calls use the backend wrapper generated beside the export.

## Public model

The authored vocabulary is intentionally small:

- `#[derive(rspyts::Type)]` on a real domain value;
- `#[derive(rspyts::Error)]` on a real domain error;
- `#[rspyts::export]` on a public function or curated inherent impl;
- `#[rspyts(buffer)]` and `#[rspyts(bytes)]` for transport policy;
- `#[rspyts(returns(buffer))]` and `#[rspyts(returns(bytes))]` for return policy;
- `#[rspyts(error = path::Error)]` when a one-parameter result alias hides its
  fixed domain error;
- `#[rspyts(wire = T)]` for an honest custom serialized shape;
- one `rspyts::module!(native)` declaration per exporting crate, optionally
  qualified with `python` or `typescript` for a single-host crate.

Python-only and executable-TypeScript-only functions may be selected where the
operation is genuinely target-specific, such as native path I/O. Target
selection cannot create two different data contracts.

Concrete exported endpoints are allowed around APIs whose original Rust
signature is not a portable public contract. For example, a generic
`open(impl AsRef<Path>)` may have a public `open_path(String)` endpoint. This is
behavioral curation, not a duplicate DTO.

## Semantic IR

IR version 4 records:

- crate and module identity;
- named structs, enums, aliases, fields, wire names, and requiredness;
- exact scalar widths, aware datetimes, option/list/map/tuple composition,
  JSON, bytes, and numeric-buffer dtype;
- explicit scalar defaults plus literal, length, and integer-minimum
  constraints;
- functions, parameters, return values, typed errors, and target scope;
- resources, multiple target-scoped constructors, target-scoped methods,
  mutability, and lifecycle;
- constants and their normalized values;
- documentation used by emitters.

Registrations are sorted before serialization. Duplicate type, resource,
function, and constant identities are errors. Unsupported or ambiguous shapes
fail validation before an output directory is replaced.

The canonical manifest has a SHA-256 semantic fingerprint. `rspyts.lock`
stores the manifest and fingerprint as deterministic, pretty-printed JSON, so
contract review compares meaning instead of generated implementation
formatting. The locked check is conservative: any meaningful difference
requires explicit acceptance.

Defaults and constraints are semantic, not emitter hints. Generated host
models expose them, the boundary normalizer applies them before Rust runs, and
Rust output is checked against the same rules. Aware chrono datetimes use an
RFC 3339 wire value, become Pydantic aware datetimes in Python, and remain
explicit RFC 3339 strings in TypeScript.

## Cross-package identity

Every definition and named reference carries the Cargo package that owns it.
When one contract exposes a type from another Rust package, the compiler keeps
that foreign owner in the probe instead of absorbing the dependency's exports
into the root package.

The consumer maps each imported Cargo owner to a committed dependency lock and
to the corresponding host package names. Resolution verifies the dependency
fingerprint and the exact referenced type/error definitions. Generated Python
and TypeScript import the defining host type; they do not emit a second model
or depend on another package's generated staging directory.

Dependency fingerprints and host mappings participate in lock schema 2 and in
the consumer fingerprint. Missing mappings, stale shapes, duplicate owner
aliases, undeclared imported owners, and dependency cycles are errors.

## Boundary values

Backend conversion uses a private semantic value model inside generated
wrappers. It preserves distinctions JSON cannot:

- signed and unsigned 64-bit integers;
- bytes versus a numeric `u8` buffer;
- numeric buffer element dtype;
- recursive sequences and string-keyed objects.

This value does not cross a generic C ABI and is not an installed runtime.
PyO3 and wasm-bindgen convert it directly to host-native values.

`bytes` becomes owned Python `bytes` or `Uint8Array`. `buffer` becomes an owned
NumPy array or the matching JavaScript typed array. Buffers preserve dtype and
element count, but are one-dimensional and do not preserve shape or strides.
Copying is part of the 0.4 contract: no host value borrows Rust memory after a
call returns.

Generated TypeScript value graphs are deeply readonly and Rust-returned plain
objects, arrays, tuples, and maps are recursively frozen. Typed arrays are
owned copies but remain mutable and unfrozen, matching JavaScript's runtime
rules for non-empty typed arrays.

Structured floats are finite. Numeric buffers preserve IEEE NaN and infinity.
`i64` and `u64` remain exact and become TypeScript `bigint`.

## Custom serialization

The derive handles a closed subset of unambiguous Serde shape rules. An
existing custom serializer can declare `#[rspyts(wire = T)]`; its real
Serialize/Deserialize implementation remains responsible for conversion and
validation.

No derive can prove arbitrary serializer code matches a declared type. The
consumer therefore owns a round-trip fixture for every declared wire shape.
rspyts rejects ambiguous customization instead of guessing or requiring a
mirror contract.

## Errors

A derived Rust enum or struct error becomes a typed host exception carrying:

- error type identity;
- variant code, or one stable type code for a struct error;
- display message.

Boundary conversion failures use a generated boundary error. Structured error
variant fields are deferred beyond 0.4 because they would recursively expand
the exported graph without a current product requirement.

## Resources

A curated inherent impl exports the real stateful Rust value as an opaque
resource. It may expose multiple constructors, including target-specific native
path and shared portable-byte factories. Generated target wrappers own the
resource. The `new` constructor is primary when present; other constructors are
named host factories. Every backend included by the resource export target must
retain at least one constructor after member target filtering.

Python exposes idempotent `close()` and context-manager cleanup. TypeScript
exposes idempotent `free()` and `Symbol.dispose`. Both reject use after
disposal. Consumers must not rely on garbage collection or host finalizers for
timely release.

A normal typed domain error does not poison a resource. A WebAssembly runtime
trap poisons the JavaScript instance because its state may be unreliable.
rspyts does not promise to catch every panic under WebAssembly's panic-abort
model.

## Backend contracts

### Python

Python uses PyO3 with `abi3-py311`. The wheel carries the native extension,
generated public package, typing, and package-specific helpers. It may use
normal host libraries selected by the contract, but never a Python rspyts
runtime. Consumer cdylibs map their fixed `python` feature to
`rspyts/python-extension`; the lower-level `rspyts/python` feature exists for
non-extension Rust tests.

### Executable TypeScript

Executable TypeScript uses wasm-bindgen and targets a real browser. The npm
artifact carries generated JavaScript, declarations, wrappers, and its WASM
asset. Node is a secondary smoke environment, not a separate native backend.
The consumer crate declares wasm-bindgen as an optional direct Cargo dependency
under its fixed `wasm` feature because wasm-bindgen has no supported crate-path
override for expanded attributes. This remains build-time Rust support, not an
npm runtime dependency.

### Static TypeScript

Static mode emits types, enums, constants, and tables. It never claims to
translate arbitrary Rust behavior. Consumers may author generic operations
over generated data or select the WASM mode.

## Build contract

`rspyts build` has one deterministic owner per consumer package:

1. Read strict `rspyts.toml` configuration.
2. Compile and load the contract under the exact backend-neutral probe features.
3. Sort, validate, and fingerprint the complete manifest.
4. Render configured host output in a fresh staging directory.
5. Render the semantic fingerprint into the staged artifacts and compile the
   selected host through PyO3 or wasm-bindgen as applicable.
6. Atomically replace `.rspyts/` or the explicit staging path.
7. Print a stable JSON build report.

Cargo default features are off unless a consumer explicitly opts in. Semantic
extraction uses one target-independent native probe. It uses
`probe-features` exactly, falling back to the common crate features only when
that key is omitted; it never unions the two sets or enables Python and WASM
together. A target-selected artifact build uses the common crate features plus
only that host's fixed feature, then compiles and stages only that host. This
keeps one manifest, fingerprint, and lock across separately scheduled Python
and TypeScript packaging jobs. Optional authored host source is copied into
staging with symlink and collision rejection before generated output is
written.

Python source mode deliberately stops before extension compilation so Maturin
can own the sole ABI-tagged extension in the final wheel. Standalone mode owns
that compilation and staging itself. A package must select one owner, never
ship both extensions.

Maturin and the consumer's TypeScript packager remain the release-artifact
packagers. They consume the reported staging directory. rspyts does not install
an npm or Python build backend and does not write generated implementation into
authored source.

## Drift prevention

Three boundaries are checked:

- Rust versus host artifacts: the staged output is rendered from the freshly
  compiled manifest and carries its fingerprint; installed-artifact smoke tests
  prove the selected host loads and behaves correctly.
- Python versus TypeScript: mappings come from one IR and a shared conformance
  corpus, including exact integers, errors, bytes, buffers, and resources.
- current versus reviewed API: `rspyts check --locked` compares the current
  semantic manifest with committed `rspyts.lock`.

This removes the need for generated-source diffs and handwritten parity tests.
Consumers still test domain behavior and smoke their installed artifacts;
rspyts owns boundary conformance.

## Deliberate exclusions

Version 0.4 is synchronous and does not support async, callbacks, streams,
iterators, generic exports, borrowed host values, arbitrary Serde, structured
error fields, zero-copy host buffers, function-body transpilation, or a generic
plugin backend. See [Limitations](../limitations.md) for the user-facing list.

## Success criterion

A consumer can change one exported Rust field, build once, and receive matching
host packages or one precise compiler error—without editing a mirror type,
conversion graph, generated source tree, runtime dependency, or custom WASM
builder.
