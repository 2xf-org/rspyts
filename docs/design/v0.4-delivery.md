# rspyts 0.4 delivery gates

The release is complete only when the compiler and its real consumer packages
pass the gates below. A later gate cannot compensate for application-authored
binding glue in an earlier one.

## 1. Compiler contract

- The only publishable implementation crates are `rspyts`, `rspyts-macros`,
  and `rspyts-cli`; fixture crates are unpublished consumers.
- The v0.3 generic C envelope, ctypes runtime, npm runtime, generated examples,
  compatibility aliases, and their implementation paths are absent.
- The macros register real types, errors, functions, resources, and constants.
- Contract loading uses explicit backend-neutral probe features, which default
  to the common consumer Cargo features.
- Cargo default features remain disabled unless explicitly opted in.
- Probe extraction never unions Python and WASM features; target-selected
  builds add and invoke only the selected host backend afterward.
- Manifest ordering, fingerprints, locks, and generated artifacts are
  deterministic.
- Unsupported types and duplicate identities fail before output replacement.
- Every generated file is below `.rspyts/` or an explicit temporary staging
  directory.

## 2. Static TypeScript

- A vocabulary fixture emits exact TypeScript types, enums, constants, and
  tables from Rust.
- `serde_json::Value` emits a recursive `JsonValue`, not `unknown`.
- Exact 64-bit integers emit `bigint`.
- Static mode emits no invented translation of Rust function bodies.
- The fixture has no npm rspyts dependency and no tracked generated source.

## 3. Python/PyO3

- A real `abi3-py311` wheel installs in a clean Python 3.11+ environment.
- Installed-wheel calls cover nested values, enum typed errors, exact 64-bit
  integers, bytes, an owned float64 NumPy buffer, scalar defaults and
  constraints, an aware datetime, and deterministic resource disposal.
- The installed wheel has no Python rspyts dependency.
- Source mode lets Maturin own the wheel's sole native extension.
- Import and behavior tests run against the wheel, not the source tree.
- Compiler/backend tests separately cover every numeric buffer dtype, struct
  errors, multiple and target-scoped resource members, return buffer/bytes
  policy, and standalone staging. Those are not described as installed-wheel
  acceptance until an artifact test exercises them.

## 4. Browser/wasm-bindgen

- A packed npm consumer artifact loads in headless Chromium.
- Installed-tarball calls cover equivalent nested values, typed errors, bytes,
  a `Float64Array`, scalar defaults and constraints, an aware RFC 3339
  datetime, and resource disposal.
- `free()` and `Symbol.dispose` are idempotent; use after disposal fails.
- A typed domain error does not poison a resource.
- The npm artifact has no rspyts dependency.
- Compiler/backend tests separately cover every numeric typed-array mapping,
  return policies, and target-scoped members.
  A WebAssembly trap is not claimed as browser acceptance until a deterministic
  real-WASM trap fixture exists.

## 5. Cross-language consumer acceptance

- Existing Rust domain types carry the derives and small transport hints.
- No consumer package contains a `Bridge*` contract or conversion graph.
- Cross-package types keep their defining package identity instead of being
  re-emitted as local models.
- Every consumer that declares a custom wire shape has a Rust round-trip
  fixture for that serializer.
- Target-scoping tests prove native-only operations do not leak into browser
  output; portable byte operations are covered by the installed host fixtures.
- Each backend has a valid constructor after target filtering, and additional
  constructors are emitted as named factories without handwritten glue.
- Python and TypeScript artifacts carry the same semantic fingerprint when they
  expose the same contract.
- Each consumer commits `rspyts.lock` and ignores `.rspyts/`.

## 6. Repository quality

- Formatting, Clippy with warnings denied, unit/integration tests, docs, and
  MSRV checks pass.
- Linux, macOS, and Windows cover the Cargo workspace; Linux builds and
  installs the reference abi3 Python wheel from a clean environment.
- A real browser covers WASM behavior.
- Archive smoke tests use the exact Cargo package candidates.
- Documentation examples are compiled or exercised.
- Searches find no tracked `.rspyts/`, v0.3 runtime dependency, generic C call
  ABI, handwritten mirror contract, or compatibility facade.

## 7. Release

- Exact candidate artifacts pass all smoke tests before tagging.
- Consumer acceptance uses those candidates before registry mutation.
- The annotated `v0.4.0` tag targets a commit already contained in `main`.
- Trusted publishing releases Cargo crates in dependency order:
  `rspyts-macros`, `rspyts`, then `rspyts-cli`.
- No PyPI or npm rspyts package is published.
- Registry artifacts are installed and smoked from a clean consumer before the
  GitHub release is created.
- A partial release is resumed from the same immutable tag; a checksum mismatch
  requires a new patch version.

The full release procedure is in [Releasing](../releasing.md).
