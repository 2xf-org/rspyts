# rspyts 0.4 delivery plan

This plan implements [direct Rust API compilation](v0.4.md) without extending
the scope while the compiler is being built.

Every phase ends with a usable vertical result. A later phase may not compensate
for a failed earlier gate with consumer-owned glue.

The implementation branch begins by deleting the v0.3 product code. Existing
code and tests are references for requirements only; they are not a migration
base and carry no presumption of reuse.

## Phase 0: freeze the contract

Deliverables:

- approve the public Rust vocabulary: `Type`, `Error`, `export`, and `module!`;
- approve `wire`, `as`, `buffer`, and `bytes` as the complete initial set of
  type escape hatches;
- define semantic IR version 4 and stable type/item identities;
- define the `rspyts.lock` format and compatibility rules;
- capture the Evaluation and Hardware source shapes used as acceptance fixtures.

Verification:

- every existing Cloud bridge type is classified as directly derivable, a
  declared wire shape, a checked scalar projection, a transport policy, or an
  intentionally different API error;
- no fixture requires a second data model;
- all unresolved cases are design changes in rspyts, not planned Cloud glue.

Exit gate: the RFC and fixture classification contain no “decide during
implementation” item that can change a public annotation or IR meaning.

## Phase 1: compiler core and static artifacts

Deliverables:

- delete the existing implementation crates, Python and TypeScript runtimes,
  generated examples, and behavior tests from the 0.4 implementation branch;
- create the three-crate 0.4 skeleton from empty files;
- implement the new derives and export registration;
- compile and load IR from the crate's actual Cargo features and `cfg` set;
- validate complete type and symbol graphs;
- implement canonical fingerprints, `inspect`, `lock`, and semantic `check`;
- write every generated file below `.rspyts/`;
- implement TypeScript static emission for types, enums, constants, and tables;
- emit a stable JSON build report.

Verification:

- repository searches find no ABI-3 envelope, ctypes runtime, v0.3 host
  runtime, or compatibility module;
- compile-fail tests cover every unsupported or ambiguous shape;
- semantic diff tests distinguish additive, compatible, and breaking changes;
- identical inputs create byte-identical IR, locks, and static artifacts;
- changing only generated text formatting does not appear as a contract change;
- generated files outside `.rspyts/` are rejected by repository fixtures.

Exit gate: a Platform-like fixture builds static TypeScript from Rust, produces
no tracked generated source, and requires no npm rspyts runtime.

## Phase 2: Python/PyO3 vertical slice

Deliverables:

- generate PyO3 abi3 registration from exported functions and impls;
- implement scalar, composite, Serde, error, and resource conversions;
- implement NumPy buffers and Python bytes from transport policy;
- generate public Pydantic models, functions, exceptions, and typing below
  `.rspyts/`;
- package a self-contained wheel through a build-only hook;
- migrate Evaluation in a fresh Cloud worktree.

Verification:

- install the wheel into a clean Python environment;
- confirm `pip` installs no rspyts runtime dependency;
- run real native calls, typed errors, and resource lifecycle tests;
- compare Rust and Python values across the shared conformance corpus;
- delete Evaluation's `bridge_contracts.rs`, committed bindings, and bridge
  conversion tests;
- import Hardware Annotation under its defining Python identity.

Exit gate: Evaluation's public Python behavior runs from existing Rust types
with zero parallel DTOs and zero tracked generated files.

## Phase 3: TypeScript/wasm-bindgen vertical slice

Deliverables:

- generate wasm-bindgen exports from the same annotated Rust surface;
- implement values, typed arrays, bytes, typed errors, resources, disposal, and
  poisoned-instance behavior;
- generate JavaScript, TypeScript declarations, initialization, and helpers
  below `.rspyts/`;
- package a self-contained npm artifact;
- migrate one narrow Hardware recording slice in the same isolated Cloud
  worktree.

Verification:

- load the package in real headless Chromium;
- run equivalent Rust, Python, and browser calls over the same values;
- cover every numeric typed-array dtype and exact 64-bit boundary;
- verify deterministic disposal and use-after-free behavior;
- verify package loading without an npm rspyts dependency;
- remove the migrated slice's handwritten bridge models and WASM staging code.

Exit gate: one real Hardware resource works in Python and the browser from the
same Rust type and methods, with no project-authored FFI glue.

## Phase 4: cross-crate identity and complete Hardware

Deliverables:

- resolve host package identities from Cargo metadata;
- generate dependency imports instead of duplicate foreign models;
- embed and verify dependency contract fingerprints;
- support every approved Hardware wire-shape and transport annotation;
- migrate the complete Hardware surface;
- remove the project-owned WASM build script.

Verification:

- Hardware's existing identifiers validate through their real Serde/domain
  implementation;
- Hardware Annotation has one Python identity and one TypeScript identity;
- native file operations are Python-only and portable byte operations match in
  Python and Chromium;
- large recordings use the intended buffer/byte paths;
- the wheel and npm artifact embed the same Hardware contract fingerprint;
- the complete Hardware release gate in the RFC passes.

Exit gate: Hardware contains no `Bridge*` data contract, `bridge_contracts`
module, committed generated binding, rspyts runtime dependency, or custom WASM
builder.

## Phase 5: Platform and clean Cloud migration

Deliverables:

- migrate Platform Python behavior to PyO3;
- migrate Platform TypeScript vocabulary to static build output;
- represent classifiers as Rust-owned tables plus generic host lookup, or use
  WASM when they are genuinely executable logic;
- regenerate the Cloud migration from current `origin/ultra-clean` rather than
  merging the old experimental branch wholesale;
- remove all v0.3 integration files and dependencies from migrated products.

Verification:

- Platform's static and native outputs share semantic fingerprints;
- every Cloud product lock has only reviewed changes;
- repository searches find no mirror graph or tracked generated implementation;
- targeted package tests pass before broad affected CI is attempted;
- clean wheel/npm installs work without rspyts host runtimes.

Exit gate: the Cloud branch is reviewable as a focused package-boundary change,
not as an accumulation of the earlier migration's unrelated work.

## Phase 6: release

Deliverables:

- publish the Rust facade and CLI as 0.4.0;
- publish the Python build hook as 0.4.0;
- publish no Python or npm runtime package required by generated clients;
- publish complete migration, architecture, packaging, and troubleshooting
  documentation;
- tag the release only after the published-artifact smoke tests pass.

Verification:

- create a clean example from the quickstart;
- build and install its wheel on every supported platform;
- build and load its npm package in Chromium and Node's WebAssembly runtime;
- rebuild Hardware and Evaluation using registry releases rather than local
  paths;
- compare artifact and lock fingerprints with the release candidates.

Exit gate: Cloud can pin 0.4.0 from registries and reproduce the artifacts used
for acceptance.

## Scope-control rules

During implementation:

- if a v0.3 internal component is proposed for reuse, reject it and implement
  the 0.4 requirement from the new design;
- if a Cloud type needs a mirror, stop and improve direct type support;
- if a host needs a project-authored codec, stop and move the capability into
  rspyts or reject the type;
- if generated implementation source must be committed, stop and repair the
  build integration;
- if a package needs the Python or npm rspyts distribution at runtime, stop and
  embed the helper;
- if a new feature is not needed by Evaluation, Hardware, or Platform, defer it
  beyond 0.4;
- if a proposed convenience weakens semantic checking, reject it.

The implementation is complete only when the difficult Hardware cases use the
same small public model as the quickstart.
